# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Mindcarter — a marketing site + role-based portal shell for a psychology/organizational
consulting business (public pages, patient/psychologist login, and a hidden admin console).
Built from a Lovable-generated TanStack Start template; this project stays connected to
[Lovable](https://lovable.dev) for sync.

## Commands

Package manager is **bun** (see `bun.lock`, `bunfig.toml`); `package-lock.json` also exists but bun is primary.

- `bun run dev` — start dev server (vite dev)
- `bun run build` — production build
- `bun run build:dev` — development-mode build
- `bun run preview` — preview a production build
- `bun run lint` — eslint over the repo
- `bun run format` — prettier --write .

There is no test suite configured in this repo.

### Package installs

`bunfig.toml` enforces a 24h supply-chain guard (`minimumReleaseAge`) blocking newly-published
package versions. Only `@lovable.dev/*` packages are excluded. Do not add new excludes without
confirming with the user first.

## Architecture

### Stack

TanStack Start (file-based SSR React router) + Vite + React 19 + Tailwind v4 + shadcn/ui
(new-york style, Radix primitives) + TanStack Query. Deploys to Vercel via `nitro/vite`
(see `vite.config.ts` — preset overridden from Nitro's Cloudflare default to `"vercel"`).

Almost all Vite plugin wiring (TanStack Start, React, Tailwind, tsconfig paths, `@` alias,
dev-only component tagger, env injection, sandbox port/host detection) is bundled inside
`@lovable.dev/vite-tanstack-config` — **do not** re-add these plugins manually in
`vite.config.ts` or the app breaks with duplicate plugins. Only pass extra config through
`defineConfig({ vite: {...} })`.

### Routing

File-based routing under `src/routes/`, per `src/routes/README.md`:

| File | URL |
| --- | --- |
| `index.tsx` | `/` |
| `about.tsx` | `/about` |
| `users/index.tsx` | `/users` |
| `users/$id.tsx` | `/users/:id` (bare `$`, no curly braces) |
| `posts/{-$category}.tsx` | `/posts/:category?` (optional segment) |
| `files/$.tsx` | `/files/*` (splat — read via `_splat`, never `*`) |
| `_layout.tsx` | layout route (renders `<Outlet />`) |
| `__root.tsx` | app shell wrapping every page |

Do **not** create `src/pages/`, `src/routes/_app/index.tsx`, or `app/layout.tsx` — those are
Next.js/Remix conventions that don't apply here. `src/routeTree.gen.ts` is auto-generated by
the TanStack router plugin — never hand-edit it.

`src/routes/__root.tsx` defines the document shell (`<html>`/`<head>`/`<Scripts>`), global
`<QueryClientProvider>`, page-transition animation (framer-motion `AnimatePresence` keyed on
pathname), and the 404 / error boundary components.

### Server error handling

SSR errors are caught at two layers so users never see a raw stack trace or blank page:

1. `src/start.ts` — a TanStack Start `requestMiddleware` that catches server-function/loader
   throws and renders `renderErrorPage()` (from `src/lib/error-page.ts`) unless the error
   already carries a `statusCode` (i.e. is an intentional HTTP error).
2. `src/server.ts` — wraps the whole Nitro fetch handler. h3 (Nitro's HTTP layer) can swallow
   in-handler throws into a generic `{"unhandled":true,"message":"HTTPError"}` 500 JSON body
   that bypasses normal try/catch — `normalizeCatastrophicSsrResponse` detects that shape and
   replaces it with the rendered error page, recovering the real error via
   `src/lib/error-capture.ts`'s out-of-band `error`/`unhandledrejection` listeners.

When touching server entry/middleware code, preserve both layers — they cover different
failure modes (thrown-but-caught vs. swallowed-by-h3).

### Auth / portals (prototype-stage, not production auth)

There are two unrelated, parallel "auth" mechanisms in this codebase — know which one a file
uses before changing it:

- **Patient / Psychologist** (`/login`, `PortalShell` in `src/components/portal-shell.tsx`):
  no real authentication. `/login` just writes a role string to `localStorage` (`mc_role`) and
  redirects; there's no credential check. Treat any "signed in" state here as UI-only.
- **Admin** (`/portal-management-access`): a hidden, unlinked route (no nav/footer links,
  blocked in `public/robots.txt`) with its own client-side login gate using a hardcoded
  email/password and a `sessionStorage`-backed session token (`__mc_adm_session`). Mock data
  (bookings, verifications, users, tickets) is inlined in the route file — there is no backend.
  `src/lib/auth.ts` defines a **separate**, unused-by-the-route HMAC session-token scheme
  (Web Crypto, no deps) intended for a real server-verified admin session — it is not currently
  wired into `/portal-management-access`, which does its own thing client-side instead.
- `/secure-admin-portal` is an intentionally dead route that always 404s — a decoy. Never link
  to it or to `/portal-management-access` from public UI (nav, footer, sitemap).
- `src/lib/auth.ts` explicitly documents that `ADMIN_EMAIL`/`ADMIN_PASSWORD`/`SESSION_SECRET`
  are placeholders to be replaced with a real DB lookup + env vars before production.

Given the above, do not assume any portal route is actually access-controlled server-side —
none of them are today.

### UI components

`src/components/ui/` is shadcn/ui-generated (new-york style, Tailwind, Radix, lucide icons —
see `components.json` for aliases: `@/components`, `@/lib`, `@/hooks`, `@/components/ui`).
Treat these as generated/vendored; prefer composing them from `src/components/` rather than
hand-editing internals unless fixing a real bug.

Hand-written layout components: `site-shell.tsx` / `site-nav.tsx` / `site-footer.tsx` (public
marketing pages), `portal-shell.tsx` (patient/psychologist portal chrome — sidebar + profile
dropdown), `page-hero.tsx`, `scroll-reveal.tsx`.

### Linting

`eslint.config.js` bans importing the `server-only` package (Next.js convention) — TanStack
Start instead uses `*.server.ts` filename suffixes or the
`@tanstack/react-start/server-only` marker for server-only modules.
